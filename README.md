# university-project
 1. Общие сведения
Название проекта: UNIVER FUTURE (Quantum System).

Назначение: Агрегатор университетов Кыргызстана с системой подбора по баллам ОРТ, сравнением вузов.

Стек технологий: * Frontend: HTML5, CSS3 (Modern Glassmorphism), JavaScript (ES6+).

Backend: Node.js (Express).

Database: PostgreSQL.

Библиотеки: Particles.js (визуальные эффекты), Bcrypt (хеширование), JWT (авторизация).

2. Цели и задачи проекта
Помочь абитуриентам быстро найти вуз по проходному баллу ОРТ.

Предоставить удобный интерфейс для сравнения стоимости обучения и рейтингов.

Обеспечить сохранность данных пользователя (избранное) через авторизацию.
3. Описание функциональных блоков
3.1. Модуль авторизации (Auth System)
Регистрация: Сбор данных (ФИО, Email, Пароль). Пароль должен хешироваться перед сохранением в БД.

Авторизация: Вход по Email/Паролю с выдачей JWT-токена.

Интерфейс: Модальное окно с переключением вкладок (Login/Reg) без перезагрузки страницы.

3.2. Информационный модуль (Каталог ВУЗов)
Главная страница: Вывод ТОП-3 университетов по рейтингу.

Фильтрация по ОРТ: Динамический расчет шансов поступления (Progress Bar) на основе введенных пользователем баллов.

Реестр: Табличный вид всех вузов для быстрого просмотра (название, город, порог, цена).

3.3. Модуль взаимодействия (User Actions)
Избранное: Возможность лайкать вузы. Состояние должно сохраняться в БД для авторизованных пользователей.

Сравнение (VS Mode): Выбор двух вузов и сравнение их характеристик в плавающей панели.

4. Технические требования
4.1. Функциональные требования (Functional Requirements)
Эти требования описывают конкретные действия, которые пользователь может совершать в системе.

Управление учетными записями:

Система должна позволять пользователю создать аккаунт (регистрация) с указанием ФИО, Email и пароля.

Система должна проверять уникальность Email (запрет регистрации на одну почту дважды).

Система должна обеспечивать вход (авторизацию) по паре Email/Пароль.

Поиск и фильтрация:

Система должна предоставлять поле для ввода баллов ОРТ.

Система должна автоматически пересчитывать «шанс поступления» для каждого вуза при изменении баллов пользователя.

Взаимодействие с контентом:

Система должна позволять пользователю добавлять университеты в список «Избранное».

Система должна сохранять список избранного в базе данных, чтобы он был доступен после перезагрузки страницы.

Система должна позволять выбирать ровно 2 объекта для режима сравнения (VS Mode).

Информационная панель:

Система должна отображать ТОП-3 университета на главной странице на основе их рейтинга.

Система должна выводить полный реестр вузов в виде таблицы с возможностью просмотра города, порога баллов и цены.

4.2. Нефункциональные требования (Non-Functional Requirements)
Эти требования описывают свойства системы: производительность, безопасность и удобство.

Производительность и отзывчивость:

Время загрузки данных из БД при первом запуске не должно превышать 2 секунд при стабильном интернете.

Фильтрация карточек и расчет шансов по ОРТ должны происходить мгновенно на стороне клиента (без перезагрузки страницы).

Безопасность (Security):

Пароли пользователей должны храниться в базе данных только в зашифрованном виде (использование алгоритма bcrypt с солью).

Доступ к защищенным действиям (например, добавление в избранное) должен осуществляться только при наличии валидного JWT-токена.

Надежность:

В случае недоступности основной базы данных (PostgreSQL), система должна отображать заранее подготовленные «демо-данные», чтобы интерфейс оставался работоспособным.

Удобство использования (Usability):

Интерфейс должен поддерживать «темную» и «светлую» темы оформления.

Система должна иметь адаптивный дизайн (корректное отображение на экранах смартфонов, планшетов и ПК).

Визуальные эффекты (частицы на фоне) не должны перегружать процессор устройства пользователя (не более 60-80 активных частиц).

Масштабируемость:

Архитектура БД должна позволять легко добавлять новые поля для университетов (например, наличие общежития или список факультетов).

5. Этапы разработки (Roadmap)
Phase 1: Разработка архитектуры БД и API (Регистрация/Вход). — Выполнено.

Phase 2: Создание фронтенда с поддержкой тем и частиц. — Выполнено.

Phase 3: Интеграция функционала сравнения и фильтрации ОРТ. — Выполнено.

Phase 4: Синхронизация «Избранного» с базой данных (в процессе).

Phase 5: Деплой на хостинг и GitHub.

6. Критерии приемки
Корректная работа переключения вкладок в модальном окне (исправление ошибки switchAuthTab).

Успешное сохранение нового пользователя в pgAdmin при регистрации.

Отображение данных из БД в карточках университетов.
